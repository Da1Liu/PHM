# UCI Hydraulic 技术路线验证方案（第一版）

> 目标：用 UCI Hydraulic System Condition Monitoring 数据集，验证 `cnc_health_baseline_technical_plan.md` 中健康基线技术路线的**可行性**，而非交付产品级 demo。
> 范围：只验证「标准化循环数据 → 特征提取 → 温度补偿 → 渐进式基线 → 异常评分 → 趋势告警 → 贡献解释」这条链路在数学和物理机制上是否成立。
> 本方案不写代码，只定义实验设计、数据切分、判据和预期结论。

---

本方案是对 `hydraulic_health_demo_steps.md` 的**数据校准版**。在真实下载并探查数据后，原 demo 文档中有四处假设需要修正（见 §2、§4），本方案以探查结果为准。

## 1. 数据集事实（已实测确认）

| 项 | 值 |
|----|----|
| 循环数（cycle） | 2205，每个循环固定 60 秒恒定负载 |
| 数据组织 | 每个传感器一个 txt，行=循环、列=循环内采样点 |
| 压力 PS1–PS6 | 100 Hz，每循环 6000 点，bar |
| 电机功率 EPS1 | 100 Hz，每循环 6000 点，W |
| 流量 FS1/FS2 | 10 Hz，每循环 600 点，l/min |
| 温度 TS1–TS4 | 1 Hz，每循环 60 点，°C |
| 振动 VS1 | 1 Hz，每循环 60 点，mm/s |
| 标签 profile.txt | 5 列：冷却器% / 阀% / 泵泄漏 / 蓄能器bar / stable flag |

退化标签分布（已确认）：

```text
泵泄漏:  0=1221  1=492  2=492
stable: 0=1449(稳定)  1=756(未达稳态)
```

**重要语义**：这些标签是**人为系统性施加**的退化等级，不是真实时间退化。`stable=1` 表示该循环可能未达稳态，应整体剔除。

## 2. 选定的验证目标变量：泵内泄漏（pump leakage）

理由：

- 在文献中泵是「中等难度」目标（冷却器/阀过于容易，蓄能器过难），最能反映真实机床健康监测的灵敏度边界。
- 与方案中液压系统「内泄漏 / 效率下降」的退化语义直接对应。
- 实测可分性充足但非平凡，适合验证算法而非验证「数据太干净」。

**可行性已实测确认**（冷却器=100 且 stable=0 子集，稳态段中段 70%）：

| 特征 | 泵0 | 泵1 | 泵2 | 可分性 sep |
|------|-----|-----|-----|-----------|
| PS1 稳态均值 | 156.54 | 156.06 | 155.75 | 6.87 |
| PS1 稳态标准差 | 6.41 | 6.37 | 6.37 | 1.74 |
| FS1 稳态均值 | 7.83 | 7.61 | 7.49 | **61.1** |
| Q/P | 0.050 | 0.049 | 0.048 | **65.6** |
| 电机功率 | 2479 | 2490 | 2503 | 3.69 |

（sep = |泵0均值 − 泵2均值| / 合并标准差，>2 即强可分）

### 修正 1：Q/P 异常方向

`hydraulic_health_demo_steps.md` 假设 Q/P 为 `high_bad`。**实测相反**：泵泄漏加重时 FS1 流量与 Q/P 均**下降**。本方案：

```text
泵泄漏目标下 Q/P 与 FS1_mean 均为 low_bad
```

并要求在方案中加入一条原则：**特征异常方向必须由基线数据 + 标签标定，不得照搬物理直觉**。

## 3. 三套实验子集（核心设计）

由于温度变化与冷却器故障强烈混淆（见修正 2），单一子集无法同时验证「检测能力」和「温度补偿」。本方案拆成三套互不冲突的子集。

### 子集 D — 检测能力（Detection）

```text
筛选条件: cooler=100 AND stable=0
泵0=169  泵1=160  泵2=160   (共 489)
温度跨度: 约 1°C（几乎恒温）
```

用途：验证特征提取、分位数评分、PCA+T²+SPE 能否区分泵泄漏等级。温度近恒定，**排除温度干扰**，纯看检测。

### 子集 T — 温度鲁棒性（Temperature robustness）

```text
筛选条件: pump=0 AND stable=0  （泵全程健康，让冷却器自由变化）
样本数: 489
温度跨度: 约 22.5°C（35~58°C）
```

用途：所有样本**泵都健康**，因此健康度理应保持平稳。但油温跨 22°C。验证：**温度补偿能否把因温度漂移产生的虚假健康度波动压下去**。这是温度补偿的关键检验，子集 D 因近恒温无法承担。

### 修正 2：温度补偿必须独立成轨

原 demo 文档计划在同一 pipeline 上做温度补偿消融。但实测：固定冷却器=100 后温度跨度仅 ~1°C，补偿无意义；放开冷却器则其故障会混入泵信号。故温度补偿**只在子集 T 上验证**，与检测能力（子集 D）解耦。

### 子集 M — 渐进式冷启动（Maturity / cold-start）

```text
来源: 子集 D 的泵0样本(169条) 作为"健康历史"按固定种子排序,
      再依次拼接 泵1、泵2 样本模拟退化发生
```

用途：验证 n<30 / 30≤n<100 / n≥100 三阶段切换时健康度曲线是否平滑、有无跳变。

## 4. 特征集（第一版，最小集）

健康特征（进入模型）：

| 特征 | 来源 | 方向 | 说明 |
|------|------|------|------|
| 稳态压力均值 | PS1 中段70% | low_bad | 泄漏使供压能力下降 |
| 稳态压力标准差 | PS1 中段70% | high_bad | 波动增大代表稳定性变差 |
| 稳态流量均值 | FS1 中段70% | low_bad | 泄漏使有效流量下降（实测） |
| Q/P | FS1/PS1 | low_bad | 实测随泄漏下降（修正方向） |

协变量（只用于回归去除，不进模型）：

| 协变量 | 来源 |
|--------|------|
| 油温 | TS1（或 TS1–TS4 均值），取循环内均值 |

### 稳态段与建压时间

- 稳态段：先用固定比例（丢前 20% 建压段 + 后 10% 收尾，取中间 70%），与现有文档一致。
- 「压力建立时间」特征**第一版暂不纳入**：UCI 循环为恒定负载、建压过程不一定清晰，且自动识别建压起点逻辑未验证。留待检测主链路跑通后再作为增强项评估（呼应技术方案中「建立时间特征提取逻辑未定义」的待办）。

## 5. 基线切分原则

对每套子集统一遵守：

```text
baseline_train      : 健康样本(泵0)前 60%   —— 拟合温度回归/分位数/PCA
baseline_calibration: 健康样本(泵0)后 40%   —— 仅校准 UCL/阈值
online_test         : 剩余健康样本 + 全部退化样本(泵1/泵2)
```

铁律：

- online_test 不参与任何训练或阈值估计。
- 所有随机操作固定种子，结果可复现。
- 温度回归只在 baseline_train 的健康样本上拟合。

## 6. 算法验证点（对应技术路线的待验证决策）

| 编号 | 技术路线中的决策 | 在哪套子集验证 | 通过判据 |
|------|------------------|----------------|----------|
| A1 | 分位数评分（冷启动期）+ 方向感知 | D | 健康样本平均健康度 > 退化样本，且 low_bad/high_bad 均生效 |
| A2 | PCA + Hotelling T² + SPE（成熟期） | D | 泵泄漏越重 score 越高；退化等级 vs score 的 Spearman 显著为正 |
| A3 | 温度协变量回归 | T | 补偿后健康段健康度方差下降；油温与残差特征相关系数趋近 0 |
| A4 | EWMA 趋势告警 | M / D | 健康段误报率 < 5%；退化段能给出首次告警位置 |
| A5 | 渐进三阶段切换 | M | 阶段切换处健康度无明显跳变（跳变则需混合过渡） |
| A6 | 特征贡献解释 | D | 泵泄漏样本的 top 贡献特征落在流量/Q-P，与物理一致 |

### 关于 PCA 主成分数（呼应技术方案讨论的过拟合风险）

第一版特征只有 4 个，PCA 用「保留 90% 方差」很可能只留 1–2 个主成分，UCL 可能过松。本方案要求：

```text
对比 固定 k=2 与 累计方差 90% 两种策略的 score 灵敏度，
在 metrics 中并列输出，不预设结论。
```

## 8. 验证步骤顺序

按依赖关系串行，不并行铺开：

1. 数据加载器：读 PS1/FS1/EPS1/TS1 + profile.txt，按 §3 三套子集筛选，剔除 stable=1。
2. 特征提取：稳态段 4 特征 + 油温协变量，输出特征分布图自检。
3. 基线切分（§5）+ 子集 D 上的分位数评分（验证点 A1、A6）。
4. 子集 D 上的 PCA+T²+SPE（验证点 A2），含 PCA 主成分数对比。
5. 子集 T 上的温度补偿消融（验证点 A3）—— 本方案最关键、风险最高的一步。
6. EWMA + 子集 M 渐进切换（验证点 A4、A5）。
7. 汇总 metrics 与图表。

## 9. 通过标准（可行性结论判据）

技术路线判定为「可行」需同时满足：

- **检测成立**：子集 D 上退化等级与 score 的 Spearman 显著为正，泵2 平均健康度明显低于泵0。
- **温度补偿有正收益**：子集 T 上补偿后健康段方差不高于补偿前（最好显著下降），且油温-残差相关系数下降。
- **误报可控**：三套子集健康段误报率均 <5%。
- **阶段切换平滑**：子集 M 上三阶段切换无破坏性跳变。
- **解释合理**：泵泄漏样本 top 贡献特征落在流量/Q-P 类，与物理机制一致。

任一项不满足，则定位是「特征设计问题 / 算法参数问题 / 数据集局限」，并在结论中明确该问题是否会迁移到真实机床场景。

## 10. 本验证能证明什么、不能证明什么

**能证明**（在液压受控循环上）：

- 标准化循环数据可转为健康基线样本。
- 温度协变量回归能否降低油温干扰（子集 T 直接检验）。
- 分位数（冷启动）与 PCA+T²+SPE（成熟期）的相对收益。
- EWMA 把单点异常转为趋势告警的有效性。
- 渐进三阶段切换的平滑性。

**不能证明**：

- 真实重型机床现场的长期误报率。
- 进给/主轴系统（振动、伺服电流）的特征设计——UCI 无对应工况语义。
- 健康度等于相对出厂新机的绝对健康水平。
- 故障部件定位能力。
- 磨合期稳定性加权——UCI 无磨合期数据，此项需换数据集（如 Ottawa Bearing）验证。

## 11. 已知局限与后续

- UCI 退化为人为施加，非真实时间序列，「模拟时序」结论须谨慎外推。
- 温度变化由冷却器驱动，与冷却器故障天然耦合；本方案用「泵全健康」子集 T 规避，但这意味着温度补偿是在「健康泵 + 变温」而非「退化泵 + 变温」上验证，外推到「退化与温度同时变化」需在真实数据上复核。
- 磨合期、工况分层、振动/电流特征等技术路线要点，UCI 无法覆盖，列入下一阶段（进给系统 PHM 2021 / 轴承 Ottawa）验证清单。
